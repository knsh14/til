# AGENT.md

本ドキュメントは、指定された **単一テーマ** について高信頼の一次情報に基づく詳細な調査と、そこから派生するポッドキャスト台本の生成を行うエージェントの実行規約である。  
読者想定は **日本の高校卒業程度の知識**。**図表は使用しない**（文字のみで理解可能にする）。

---

## 0. 目的（Deliverables）

1. **長文レポート（一次情報ベース）**  
   - 出力先: `./doc/<slug>.md`  
   - 文字数制約なし。指定テーマを **体系的・網羅的** にまとめる。  
   - 出典は **大学講義資料 / 公式研究機関 / 学術誌（査読済み含む）** などの一次情報のみ。  
   - 読者が段階的に理解できるように、**3〜5 個の最重要トピック**を骨子として詳細に展開する。

2. **ポッドキャスト台本**  
   - 出力先: `./transcript/<slug>.md`  
   - `doc/<slug>.md` を基に作成（内容の忠実な変換＋口語化）。  
   - 説明は高校卒業レベルで平易に。各セグメントは 3〜6 分想定。  
   - ナレーター/ホスト/ゲスト等の役割、SE/間（pause）の指示、章立て、要約、結論、参考文献読み上げの方針を含む。

---

## 1. ディレクトリとファイル命名規則

- `doc/` と `transcript/` は必須。存在しない場合は作成する。  
- `<slug>` はテーマの日本語タイトルから自動生成（小文字、半角英数字とハイフン、先頭末尾のハイフン除去）。  
  - 例: テーマ「量子コンピュータの基礎と応用」 → `ryoshi-computer-kiso-ouyou`

**ファイルパス例**
- `doc/ryoshi-computer-kiso-ouyou.md`
- `transcript/ryoshi-computer-kiso-ouyou.md`

---

## 2. 出典と情報源ポリシー（厳守）

- **使用可（一次情報・高信頼）**  
  - 大学の公式講義資料・シラバス・コースノート  
  - 公式研究機関・国際機関（例：NIMS, RIKEN, NICT, NEDO, 国土地理院, WHO, OECD, NASA, ESA 等）の一次資料  
  - 学術誌（査読済み論文、レビュー論文、特集号）・学会プロシーディング  
- **原則不使用**  
  - まとめブログ、個人ブログ、SNS、匿名掲示板、生成 AI 出力、出典不明の資料  
  - 二次・三次資料は、一次情報に到達できない補助的事情がある場合のみ「補助」として参照可能だが、本文の主要根拠にはしない。

**出典表記スタイル**
- 本文中：脚注風の角括弧番号 `[1]` を付し、文末に「参考文献」節で完全情報を列挙。  
- 文献情報：著者名（または機関名）、年、タイトル、媒体（ジャーナル/機関）、巻号またはレポート番号、ページ/DOI/恒久的識別子。  
- オンライン資料は可能なら DOI または恒久リンク。アクセス日を付す。

---

## 3. 調査・執筆ワークフロー

1. **テーマ受領**：`テーマ: <日本語>`  
2. **骨子設計**：テーマを分解し、**3〜5 の最重要トピック**を決定。選定基準は「理解に必須」「最新の合意形成がある」「応用上の重要性」。  
3. **一次情報収集**：上記ポリシーに合致する資料のみ採用。  
4. **検証**：重要数値・定義・主張は最低 2 つ以上の一次情報でクロスチェック。  
5. **構成**：定義 → 背景 → 主要理論/メカニズム → 実例/データ → 論争点 → 応用/影響 → 限界/課題 → 展望 の順を基本に、**因果関係が追える**論理を構築。  
6. **執筆（doc）**：図無しで理解できるよう、説明→要点→用語解説を段落で積み上げる。  
7. **品質確認**：下記チェックリストに合格するまで改稿。  
8. **台本化（transcript）**：`doc` を口語化・段取り化し、時間配分・役割・要約・結論・CTA を付与。  
9. **最終点検**：発音しづらい用語にルビ（読み）候補、専門語は簡潔に直訳＋意訳を用意。

## 4. レポート（doc）仕様

### 4.1 フロントマター（YAML）
```yaml
---
title: "<日本語タイトル>"
slug: "<slug>"
topic: "<テーマ原文>"
language: "ja"
audience: "JP-highschool-grad"
created: "<YYYY-MM-DD>"
updated: "<YYYY-MM-DD>"
source_policy: "primary-only"
keywords: ["…", "…"]
word_count: <自動算出可>
---
```

### 4.2 構成テンプレート（文字のみ）
```
# タイトル

## 0. 概要（Executive Summary）
- 目的・結論の先出し（5–10 行）

## 1. 主要トピック一覧（3〜5）
1) …
2) …
3) …
(4) …
(5) …

## 2. 前提知識と用語
- 高校卒業レベルの範囲で定義・背景を丁寧に
- 必要なら式は言語で説明（記号の羅列は避ける）

## 3. トピック1：<見出し>
- 定義／仕組み
- 代表的な一次研究・データ（本文内で根拠提示 [1][2]）
- 重要な数値やレンジは出典とセットで
- 応用・影響・限界

## 4. トピック2：<見出し>
（同上）

## 5. トピック3：<見出し>
（同上）

## 6. 関連・統合
- トピック間の相互関係、因果、トレードオフ

## 7. 反論可能性と現在の論争点
- 異説・限界・未解決問題

## 8. 実社会への影響・実装上の留意点
- 経済・倫理・法規制・教育等

## 9. まとめと今後の課題
- 要点の再確認
- 中長期の見通し

## 参考文献
[1] …
[2] …
```

> **注**：別要件として「18000 字程度のまとめ」を必要とする場合は、本 `doc` とは別に `summary/` を設けても良いが、通常は `doc` 本文を十分に厚く書く。

## 5. ポッドキャスト台本（transcript）仕様

### 5.1 フロントマター（YAML）
```yaml
---
title: "<日本語タイトル>（Podcast Script）"
slug: "<slug>"
source_doc: "doc/<slug>.md"
language: "ja"
audience: "JP-highschool-grad"
duration_min: 20-40
roles: ["Host", "Narrator", "Guest(任意)"]
---
```

### 5.2 台本テンプレート（口語＋指示）
```
# オープニング（約1–2分）
HOST: こんにちは、○○です。今日は「<タイトル>」を、高信頼の一次情報に基づいてわかりやすく解説します。

# セクションA：導入（約3分）
NARRATOR: 本編の前に前提となる用語を整理します。…（高校卒業レベルで噛み砕く）

# セクションB：主要トピック1（約4–6分）
HOST: まず最初のポイントは「…」です。
NARRATOR: これは[1][2]で示されており、要するに…（数値や結論は具体的に）

# セクションC：主要トピック2（約4–6分）
（同様）

# セクションD：主要トピック3（約4–6分）
（同様。4〜5個まで拡張可）

# セクションE：統合・影響（約3–5分）
HOST: ここまでの話をつなげると…
NARRATOR: 実社会では… 倫理・法・教育の観点では…

# クロージング（約1–2分）
HOST: 今日は「…」を扱いました。詳しい参考文献は概要欄（またはエピソードノート）をご覧ください。
NARRATOR: 参考文献読み上げ（主要 3–5 件、DOI/機関名を簡潔に）
```

**音声向け最適化**
- 専門用語は最初の出現時に「短い定義」→繰り返し時は略称。  
- 長い文は 20〜25 音節程度で区切る。  
- 難しい固有名はカタカナ読みを括弧で補足（例：シュレディンガー）。  
- 数値は「桁区切り」「比較対象」を添えて提示。

---

## 6. 品質チェックリスト（両成果物共通）

- [ ] 3〜5 の最重要トピックが明確で、相互の関係が説明されている  
- [ ] 主要主張・数値は **最低 2 本**の一次情報でクロスチェック  
- [ ] 参考文献は **一次情報のみ**（ブログ・SNS無し）  
- [ ] 図表不使用でも段階的に理解できる説明になっている  
- [ ] 高校卒業レベルに配慮し、専門用語へ簡潔な定義を付与  
- [ ] 反論可能性・限界・未解決点の記述がある  
- [ ] `doc` → `transcript` への内容トレース（齟齬なし）  
- [ ] フロントマター／メタデータが充足  
- [ ] 誤字脱字・日本語の可読性・冗長表現の整理済み

---

## 7. 執筆スタイル・トーン

- **明晰・具体・検証可能**。比喩は必要最小限。  
- 強い断定は **一次情報の合意** がある場合のみ。その他は「現時点の知見では…」等の準保留を用いる。  
- 主張 → 根拠 → 含意 → 例示 の順で短い段落を積み上げる。  
- 「結論の先出し」と「章末のミニ要約」を徹底。

---

## 8. 実行インターフェース（プロンプト雛形）

**実行時入力（例）**
```
テーマ: 量子コンピュータの基礎と応用
要件: 一次情報のみ、図表禁止、日本語、高校卒業レベル
```

**AGENTの動作**
1) テーマを受理して `<slug>` を生成  
2) 骨子（3〜5 トピック）を決めて明示  
3) 収集→検証→`doc/<slug>.md` 執筆  
4) `transcript/<slug>.md` を `doc` から生成  
5) チェックリストで検収し、完了を報告（ファイルパスを明記）

---

## 9. よくある落とし穴（回避策）

- **二次資料の紛れ込み**：必ず DOI/機関ページに遡及し一次出典を確認。  
- **専門用語の未定義**：最初の登場時に1行定義。  
- **数値の単位抜け**：値＋単位＋測定条件をセットで記載。  
- **冗長化**：章末に 3 行の要点サマリで引き締める。  
- **口語化の歪み**：`transcript` では比喩・例えを足すが、根拠のニュアンスは保持。

---

## 10. ライセンス・倫理

- 著作権に注意。図表転載はそもそも行わない。引用は短く、公正使用の範囲内で。  
- 被験者データ等の扱いは、出典機関のガイドラインに従う。  
- 偏りのある主張は、反対仮説や代替説明の可能性も併記してバランスを保つ。
